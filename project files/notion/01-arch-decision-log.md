## ADR-001: استفاده از معماری RAG به جای Fine-tuning مدل زبانی

**تاریخ:** ۱۲ خرداد ۱۴۰۴ (June 1, 2026)
**وضعیت:** ✅ تصویب‌شده
**تصمیم‌گیرندگان:** امیرمسعود آشوری، مجیدرضا کریمی

### پس‌زمینه

پروژه نیاز به یک سیستم پرسش‌وپاسخ تخصصی در حوزه مهارت‌های بقا و مدیریت بحران دارد. این حوزه شامل اطلاعات دقیق و ساختارمند است که باید پاسخ‌ها دقیقاً بر اساس این اطلاعات باشد و نه بر اساس دانش عمومی مدل. دو رویکرد اصلی وجود داشت:

1. **Fine-tuning**: آموزش مجدد یک مدل زبانی بر روی داده‌های بحران
2. **RAG (Retrieval-Augmented Generation)**: بازیابی اطلاعات از پایگاه دانش و تولید پاسخ

### گزینه‌های بررسی‌شده

| معیار | Fine-tuning | RAG |
|-------|------------|-----|
| نیاز به GPU | بله — حداقل A100 | خیر — CPU کافی است |
| زمان آموزش | ۱۲–۴۸ ساعت | صفر (فقط indexing) |
| قابلیت به‌روزرسانی دانش | نیاز به آموزش مجدد | اضافه کردن یک فایل MD |
| قابلیت توضیح (Explainability) | پایین | بالا — منابع قابل استناد |
| خطر Hallucination | بالا | پایین — پاسخ‌ها از متن کتاب |
| هزینه | بالا | پایین |

### تصمیم

**RAG** با stack زیر انتخاب شد:
- **Vector DB:** Qdrant (local)
- **Embedder:** multilingual-e5-base
- **Reranker:** bge-reranker-v2-m3
- **LLM:** Qwen2.5-7B-instruct (Ollama) / قابل تعویض با هر endpoint سازگار با OpenAI

### نتایج مثبت
- پایگاه دانش در هر زمان قابل بروزرسانی است بدون آموزش مجدد
- پاسخ‌ها دارای منبع (source citations) هستند
- قابل اجرا روی سخت‌افزار معمولی (Windows 10 + CPU)
- هر LLM سازگار با OpenAI قابل جایگزینی است

### ریسک‌ها / محدودیت‌ها
- کیفیت پاسخ به کیفیت retrieval وابسته است
- LLM محلی (Qwen 7B) فارسی ضعیف‌تری نسبت به مدل‌های frontier دارد
- نیاز به سه سرویس مجزا (Qdrant + Ollama + FastAPI)

## ADR-002: انتخاب Qdrant به عنوان پایگاه داده برداری

**تاریخ:** ۱۲ خرداد ۱۴۰۴ (June 1, 2026)
**وضعیت:** ✅ تصویب‌شده
**تصمیم‌گیرنده:** مجیدرضا کریمی

### پس‌زمینه

پروژه به یک vector database نیاز داشت که روی Windows 10 بدون Docker قابل اجرا باشد، داده‌ها را روی دیسک ذخیره کند، و یک HTTP API ساده ارائه دهد.

### گزینه‌های بررسی‌شده

| معیار | FAISS | Chroma | Pinecone | **Qdrant** |
|-------|-------|--------|---------|---------|
| نوع | In-memory | In-process | Cloud | Standalone |
| Persistence | ❌ (باید دستی Save کرد) | ✅ | ✅ | ✅ |
| HTTP API | ❌ | ❌ | ✅ | ✅ |
| نیاز به Docker | ❌ | ❌ | ❌ | ❌ (باینری مستقل) |
| اجرا روی Windows | ✅ | ✅ | N/A | ✅ |
| هزینه | رایگان | رایگان | پولی | رایگان |
| عملکرد با ۱۶۷ chunk | کافی | کافی | کافی | عالی |
| فیلتر بر اساس Metadata | محدود | محدود | ✅ | ✅ |

### تصمیم

**Qdrant** به دلایل زیر انتخاب شد:
1. باینری مستقل — بدون Docker، نیاز به `qdrant.exe` و اجرا از command line
2. HTTP API روی پورت 6333 — ادغام آسان با FastAPI
3. ذخیره‌سازی دائمی در پوشه `tools/storage/`
4. پشتیبانی از فیلتر metadata برای آینده (فیلتر بر اساس موضوع بحران)

### نتایج
- اجرا پایدار بدون هیچ وابستگی اضافه
- حذف یا بازسازی index با یک دستور `python -m scripts.ingest`
- لاگ خوانا و debugging آسان

## ADR-003: انتخاب multilingual-e5-base برای تبدیل متن به بردار

**تاریخ:** ۱۳ خرداد ۱۴۰۴ (June 2, 2026)
**وضعیت:** ✅ تصویب‌شده (جایگزین انتخاب اولیه)
**تصمیم‌گیرنده:** مجیدرضا کریمی
**توجه:** این تصمیم جایگزین تلاش اولیه با `paraphrase-multilingual-mpnet-base-v2` شد

### پس‌زمینه

Embedder باید متن فارسی و انگلیسی را به‌خوبی پردازش کند. پایگاه دانش پروژه به فارسی نوشته شده، اما کاربران ممکن است به فارسی یا انگلیسی سوال بپرسند.

در تلاش اولیه از `sentence-transformers/paraphrase-multilingual-mpnet-base-v2` استفاده شد که کیفیت retrieval فارسی آن در تست‌های اولیه ضعیف بود.

### گزینه‌های بررسی‌شده

| مدل | بُعد بردار | فارسی | انگلیسی | اندازه | یادداشت |
|-----|-----------|-------|---------|------|---------|
| paraphrase-multilingual-mpnet | 768 | متوسط | خوب | 278MB | انتخاب اولیه — کیفیت کافی نبود |
| OpenAI text-embedding-ada-002 | 1536 | خوب | عالی | API | هزینه دارد، وابسته به اینترنت |
| **intfloat/multilingual-e5-base** | **768** | **خوب** | **خوب** | **278MB** | **انتخاب نهایی** |
| all-MiniLM-L6-v2 | 384 | ضعیف | خوب | 80MB | فارسی ندارد |

### تصمیم

**intfloat/multilingual-e5-base** به دلایل زیر انتخاب شد:
1. آموزش دیده روی ۱۰۰+ زبان از جمله فارسی
2. پشتیبانی از `query:` و `passage:` prefix برای semantic search بهتر
3. سایز ۲۷۸MB — قابل اجرا روی CPU
4. Benchmark بهتر روی BEIR در مقایسه با گزینه‌های مشابه
5. رایگان و offline — بدون وابستگی به API

### تغییر در فرآیند Ingestion

متن‌های query با prefix `"query: "` و passage‌ها با `"passage: "` embed می‌شوند تا دقت cosine similarity افزایش یابد.

## ADR-004: افزودن Reranker برای فیلتر دامنه‌ای و بهبود دقت

**تاریخ:** ۲۵ خرداد ۱۴۰۴ (June 14, 2026)
**وضعیت:** ✅ تصویب‌شده
**تصمیم‌گیرنده:** مجیدرضا کریمی، امیرمسعود آشوری
**محرک:** سوالات خارج از دامنه پاسخ می‌گرفتند

### پس‌زمینه

پس از اتمام پایپ‌لاین RAG اولیه، در تست‌ها مشاهده شد که سوالات کاملاً غیرمرتبط (مثل «رستوران خوب تهران؟» یا «ریاضی بخوانم یا زبان؟») نیز پاسخ می‌گرفتند. این به دلیل ماهیت cosine similarity بود که همیشه نزدیک‌ترین بردار را برمی‌گرداند حتی اگر ربطی نداشته باشد.

دو روش برای حل این مشکل بررسی شد:
1. **Threshold روی vector similarity** — محدودیت: فاصله در فضای برداری همیشه معنادار نیست
2. **Reranker (Cross-encoder)** — دقیق‌تر: متن سوال و passage را با هم ارزیابی می‌کند

### گزینه‌های بررسی‌شده

| روش | دقت | سرعت | پیچیدگی |
|-----|-----|------|---------|
| Min vector score threshold | متوسط | سریع | کم |
| BM25 lexical filter | متوسط | سریع | متوسط |
| **BAAI/bge-reranker-v2-m3** | **بالا** | **متوسط** | **متوسط** |
| Cohere Rerank API | بالا | خوب | کم — اما هزینه دارد |

### تصمیم

**BAAI/bge-reranker-v2-m3** (Cross-encoder reranker) با threshold پیکربندی‌پذیر:
- `domain.min_rerank_score: 0.02` (در `configs/config.yaml`)
- سوالات in-domain: score ≥ 0.08
- سوالات off-topic: score ≈ 0.00
- شکاف واضح → فیلتر تمیز

### نتایج
- نرخ پاسخ به سوالات خارج دامنه به صفر رسید
- کیفیت پاسخ‌های داخل دامنه بهبود یافت (top-3 به جای top-10)
- زمان پاسخ حدود 500ms اضافه شد (قابل قبول)
- threshold از طریق config قابل تنظیم است بدون تغییر کد

## ADR-005: انتخاب FastAPI برای لایه Backend

**تاریخ:** ۱۲ خرداد ۱۴۰۴ (June 1, 2026)
**وضعیت:** ✅ تصویب‌شده
**تصمیم‌گیرنده:** محمد مجتبی کچویی، امیرمسعود آشوری

### پس‌زمینه

پروژه به یک API server نیاز داشت که:
- اندپوینت POST /chat را پشتیبانی کند
- با کتابخانه‌های ML پایتون (sentence-transformers، qdrant-client، openai) ادغام شود
- سرعت توسعه بالایی داشته باشد
- داکیومنتیشن خودکار تولید کند

### گزینه‌های بررسی‌شده

| معیار | Django | Flask | **FastAPI** |
|-------|--------|-------|---------|
| Async support | محدود | ❌ | ✅ Native |
| Auto OpenAPI docs | ❌ | ❌ | ✅ /docs |
| Type hints + Pydantic | ❌ | ❌ | ✅ Native |
| سرعت توسعه | کند | متوسط | سریع |
| Overhead | بالا (ORM، admin) | کم | کم |
| مناسب برای ML API | خیر | بله | بله (بهتر) |

### تصمیم

**FastAPI** با Uvicorn ASGI server. دلایل:
1. Pydantic validation — تایپ‌چک خودکار ورودی/خروجی
2. `/docs` — Swagger UI رایگان برای تست API
3. async/await — آماده برای I/O سنگین (مدل‌ها، HTTP به Ollama)
4. سرعت توسعه — 3 اندپوینت در یک روز کامل شد

### ساختار نهایی
```
app/
├── server.py     # FastAPI app، روتر، CORS، startup
├── rag.py        # کل پایپ‌لاین RAG
└── static/
    └── index.html  # Frontend
```

## ADR-006: رویکرد Offline-First برای ابزارهای نجات اضطراری

**تاریخ:** ۱ تیر ۱۴۰۴ (June 21, 2026)
**وضعیت:** ✅ تصویب‌شده
**تصمیم‌گیرنده:** امیرمسعود آشوری

### پس‌زمینه

در طراحی Sprint 3، سه ابزار اضطراری به پروژه اضافه شد:
- **حالت اضطرار (Panic Mode):** راهنمای گام‌به‌گام بحران
- **جعبه‌ابزار نجات (SOS Toolkit):** آژیر، Morse strobe، آدرس GPS
- **کیت اضطراری:** چک‌لیست کوله پشتی اضطراری

سوال اصلی: آیا این ابزارها باید به backend وابسته باشند؟

### تحلیل

**بحران = قطعی احتمالی اینترنت.** اگر کاربر در زلزله یا آتش‌سوزی باشد، احتمال بالاست که:
- اینترنت قطع باشد
- Backend در دسترس نباشد
- کاربر به ابزار فوری و بدون تأخیر نیاز داشته باشد

### تصمیم

**تمام ابزارهای نجات کاملاً client-side هستند:**
- Panic Mode: اطلاعات hard-code شده در JavaScript
- SOS: WebAudio API (آژیر/سوت)، CSS Animation (Morse strobe)، Geolocation API (GPS)
- Emergency Kit: SVG inline (قابل دانلود)، localStorage برای چک‌لیست

**هیچ فراخوانی API در این بخش‌ها وجود ندارد.**

### نتایج
- ابزارهای نجات حتی بدون backend، اینترنت، یا سرور کار می‌کنند
- تنها تب «گفتگو» به backend نیاز دارد
- کاربر می‌تواند صفحه را bookmark کند و آفلاین استفاده کند

## ADR-007: Frontend تک‌فایلی با Vanilla JavaScript

**تاریخ:** ۱۵ خرداد ۱۴۰۴ (June 4, 2026)
**وضعیت:** ✅ تصویب‌شده
**تصمیم‌گیرنده:** امیرمسعود آشوری، محمد مجتبی کچویی

### پس‌زمینه

پروژه به یک رابط کاربری وب با چهار تب، پشتیبانی RTL، و تم فارسی تاریک نیاز داشت. سوال: آیا از یک framework مدرن استفاده کنیم؟

### گزینه‌های بررسی‌شده

| معیار | React/Vue/Angular | **Vanilla JS** |
|-------|-----------------|-------------|
| نیاز به Build step | بله (npm build) | ❌ ندارد |
| وابستگی‌های خارجی | بسیار زیاد | صفر |
| کار آفلاین | نیاز به bundled version | ✅ نیتیو |
| یادگیری تیم | متوسط/سنگین | ✅ همه آشنا |
| استقرار | نیاز به CDN یا build | ✅ فقط یک فایل |
| اندازه | ۱۰۰KB+ (حتی با tree-shaking) | < 5KB |

### تصمیم

**یک فایل HTML تک‌فایلی (`app/static/index.html`)** با:
- Vanilla JavaScript (ES6+)
- CSS inline (RTL، تم تاریک، متغیرهای رنگ)
- بدون هیچ وابستگی npm یا CDN

### نتایج
- استقرار: کپی یک فایل کافی است
- آفلاین: کاملاً کار می‌کند
- FastAPI مستقیماً این فایل را serve می‌کند (`StaticFiles`)
- حجم کل frontend: < 50KB

### یادداشت
این تصمیم برای یک پروژه دانشگاهی با deadline کوتاه کاملاً مناسب است. در صورت توسعه به پروژه واقعی، migration به React/Vue توصیه می‌شود.
